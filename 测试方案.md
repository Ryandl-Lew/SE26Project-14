## 一、测试概述

### 1.1 测试目标

| 目标 | 说明 |
|------|------|
| 功能正确性 | 所有页面功能与 API 接口按预期工作，覆盖 8 个核心用例 |
| 界面一致性 | 静态原型中 10 个页面的 UI 布局、交互逻辑在实际实现中保持一致 |
| 性能达标 | ≥100 并发用户，核心操作平均响应时间 ≤3 秒（Agent 分析除外） |
| 数据可靠性 | 支持 ≥10,000 条记录，草稿崩溃恢复，数据不丢失 |
| 兼容性 | Chrome / Edge 浏览器正常运行，响应式三档断点适配正确 |

### 1.2 测试范围界定

**包含：**
- 功能测试（10 个页面 + 70 个 API 接口）
- 界面 / 布局一致性测试
- 兼容性测试（浏览器 + 响应式三档断点）
- 性能测试（100 并发压测、大数据量 ≥10k 记录）
- 安全性测试（认证鉴权、注入防护、文件上传校验）

**不包含（本次迭代）：**
- 进阶需求（图片识别自动填表、多人协作变更日志）
- 移动端 App 测试（当前仅 Web 浏览器）

## 三、测试策略

### 3.1 测试金字塔

```
        ╱ E2E 测试 ╲         关键业务流程（5 条核心流程）
       ╱────────────╲
      ╱  集成测试      ╲       前后端接口联调、模块间数据流转
     ╱────────────────╲
    ╱   单元测试          ╲    组件渲染、工具函数、API 处理逻辑
   ╱──────────────────────╲
```

### 3.2 各层测试说明

| 测试层级 | 范围 | 工具 | 覆盖目标 |
|----------|------|------|----------|
| **单元测试** | 前端组件、工具函数、后端 Service 层 | Vitest（前端）、Jest（后端） | 关键逻辑覆盖率 ≥ 60% |
| **集成测试** | API 端点 + 数据库操作 | Supertest + 测试数据库 | 所有 P0 API 100% 覆盖 |
| **E2E 测试** | 关键业务流程端到端 | Playwright | 5 条核心流程 |
| **UI 一致性测试** | 实现与原型对比 | 人工走查 + 截图对比 | 10 个页面全覆盖 |
| **性能测试** | 并发压测 + 大数据量 | k6 / Apache Bench | 100 并发 + 10k 记录 |
| **安全测试** | 认证鉴权、注入、上传 | 人工 + OWASP ZAP | P0 级别 |

## 七、缺陷管理

### 7.1 缺陷严重等级

| 等级 | 定义 | 示例 | 处理时限 |
|------|------|------|----------|
| **L1 - 致命** | 系统崩溃、数据丢失、安全漏洞 | 登录绕过、SQL 注入成功、数据丢失 | 即时修复 |
| **L2 - 严重** | 核心功能不可用 | 新建项目报错、提交审核失败 | 4 小时内 |
| **L3 - 一般** | 功能部分异常，有替代方案 | 筛选功能不正确、导出格式错乱 | 24 小时内 |
| **L4 - 轻微** | UI 样式问题、文字错误 | Badge 颜色不对、间距偏差 | 可延后 |

### 7.2 缺陷处理流程

```
发现 → 提交（标题+步骤+截图+日志） → 确认为 Bug？ → 分配修复人 → 修复 → 验证 → 关闭
                                      ↓ 否
                                   标记为"非问题"或"设计如此"
```

### 7.3 发布标准

- L1 / L2 缺陷必须全部修复
- L3 缺陷修复率 ≥ 90%
- 所有 E2E 核心流程通过
- 性能指标全部达标
